HIGH · Cards

Five card systems

.race-card · .driver-card · .listing-card · .blog-card · .rec-card · .stat

Every card has its own border-left rule, hover effect, arrow position, and (worst) different border-radius. The records card is the only one that feels alive.

As it is — five variants
.driver-card
3px left, 0 radius
.listing-card
1px border, 8 px radius
.blog-card
3 px accent left, 0 radius
.rec-card
top accent, 0 radius

Different rails, different hover-borders, different radii — the user reads it as four unrelated card types.

Fix — one base + 3 modifiers
.card .card-team
3 px team strip
.card .card-accent
top rule, var-driven colour
.card
plain — no rail
.card .is-next
accent border + glow

One radius (0), one hover (border → line-3 or accent), one arrow position (top-right). Variants only change where the colour lands.

HIGH · Radius

Mixed border-radius

.listing-card 8 px · .theme-switcher 6 px · everything else 0

f1gures is a hard-corner system — every panel, callout, pill and chart uses 0. Two outliers (listing card, theme switcher, sort buttons) carry an 8 / 6 px radius. Either kill the radius, or apply it consistently. Pick zero — it's the dominant rule.

As it is — radius mismatch
.listing-card (8 px)
.race-card (0 px)

Two cards on the same page (the home screen embeds both) read as different components.

Fix — unify on 0 px
.listing-card (0 px)
.race-card (0 px)

Keep border-radius: 50% for dots, and segmented-control rounding (6 px) if needed for trigger affordance — those are the documented exceptions.

HIGH · Inline links

Two link affordances

.inline-link (dotted border-bottom) vs :where(.data-table a …) underline override

The original .inline-link drew a 1 px dotted line-2 border-bottom. A later override in the "CONTENT LINK AFFORDANCE" section forces those same selectors to use a real underline. Both rules currently apply, fighting via !important. Net: links inside .data-table render correctly, but .inline-link elsewhere keeps the legacy dotted style on hover.

As it is — two patterns coexist

Legacy: Lewis Hamilton (dotted)

Override: Lewis Hamilton (real underline, 35 % currentColor)

Conflict resolved with !important — fragile. Hover state diverges across surfaces.

Fix — one .inline-link, true underline

Default: Lewis Hamilton

Hover: 1 px → 2 px thickness, color → currentColor.

Replace the border-bottom: 1px dotted rule in the base, drop the !important overrides. Cards/rows still get the top-right ↗ icon — that affordance is separate.

MED · Affordance

Card arrow + chevron overlap

.race-card-link ::after (↗) · .tbl tr.clickable td:last-child ::after (›)

Cards get a top-right ↗ box-arrow; clickable rows get a right-aligned › chevron. Records cards have both — the .rec-card has a custom .rec-card-arrow and would inherit the generic card ↗ if it got the .f1-card-link class. Document the exclusivity rule.

As it is — easy to double-up

An author can apply .race-card.race-card-link.f1-card-link by accident — two ↗ arrows render. Records cards override with their own arrow but still rely on a separate class.

Fix — one navigates-class

Replace the four ::after openers with a single [data-nav] attribute selector. [data-nav="card"] → ↗ top-right; [data-nav="row"] → › right-center. No more class accumulation.

MED · Type

Title case mixed with uppercase

.race-name UPPER · .blog-card-title sentence · .records-page h1 UPPER

Display headings are uppercase across the system — except blog titles and article H2/H3. The exception is intentional (long-form reading) but undocumented. Codify the rule and the system reads consistently.

As it is — no documented rule
CIRCUIT GILLES VILLENEUVE
How Imola became the season's pivot
Fix — codify "editorial = sentence"

Add the rule to Foundations: uppercase display for chrome and data screens; sentence case for blog/MDX where readability beats branding. .blog-article-body h2/h3 already implement this.

MED · Color

Ferrari ≈ accent

Ferrari #E80020 vs --accent #E8002D

A 3-byte hex away. On a Ferrari driver page, the accent rule and the team strip read as the same colour — the "active" cue collapses. Same story for Sauber vs --pos, and the Williams / Racing Bulls pair.

As it is — accent disappears
P 1 · C. Leclerc · Ferrari
Fix — see Teams · Collisions

On a Ferrari-context page, the active state shifts to a 2 px underline + weight 800. Strip keeps Ferrari red. Documented in Teams · Collisions & guardrails.

LOW · Tokens

Spacing not tokenised

Inline pixel values in app.css

Padding and gap values are inlined throughout app.css — 8 / 12 / 14 / 16 / 20 / 24. Already on a 4-unit scale; promoting them to --sp-N tokens (see Foundations) makes refactors safer.

As it is — magic numbers
.race-card { padding: 16px; gap: 12px; } .driver-card { padding: 14px; gap: 12px; }
Fix — tokenise
:root { --sp-3: 12px; --sp-4: 14px; --sp-5: 16px; } .race-card { padding: var(--sp-5); gap: var(--sp-3); }
LOW · Motion

Inlined durations

.12s · .15s · 120ms · 150ms · 300ms scattered

Three different durations in .race-card alone. Pull to --mo-fast, --mo, --mo-slow. Tiny refactor, large legibility win.

As it is
.race-card { transition: border-color .12s, background .12s; } .listing-card { transition: border-color 0.15s, box-shadow 0.15s; } .blog-card img { transition: transform 300ms; }
Fix
:root { --mo-fast: 80ms; --mo: 120ms; --mo-slow: 300ms; } .race-card { transition: border-color var(--mo), background var(--mo); }
LOW · Charts

Recharts uses literal colours

StandingsCommon.jsx · PointsChart / TeamProgressionChart

The progression charts already pull team colour for line stroke — but axis/tick/grid colours are hardcoded in the Recharts overrides. Move them to the existing --fg-3 / --line-1 tokens (they're partially there).

Partial — already uses some tokens

.recharts-cartesian-axis-tick text { fill: var(--fg-3) !important; } — good. But tooltip border is literal var(--line-2), not consistent.

Fix — finish the migration

Switch tooltip background to --bg-3, border to --line-2, label color to --fg-2 across the board. One-line change.

Migration order

Three PRs, lowest-risk first. Run the visual-regression checks against the records page, the home screen, the calendar grid, the standings table and the historical winners table — these surface every primitive.
