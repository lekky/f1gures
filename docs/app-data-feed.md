# Mobile app data feed — `/data/app/v1/`

The native mobile apps ([github.com/lekky/figures-app](https://github.com/lekky/figures-app), Android + iOS) consume a versioned JSON feed generated by this repo. **The apps ship no scoring rules and no editorial copy — everything they show comes from this feed**, so shipped app builds keep working and stay fresh as long as the contract below holds.

## What gets generated

`scripts/build-app-feed.mjs` (last step of `prebuild`, after `build:archive`) writes to `public/data/app/v1/` (gitignored; deployed via `dist/`):

| File | Contents | Size |
|---|---|---|
| `manifest.json` | schemaVersion, generatedAt, latestSeason, per-file sha256 hashes, minAppVersion, notice | ~1 KB |
| `seasons/<year>.json` (2020…current) | Per-season app contract: `meta`, `drivers[]`, `teams[]`, `calendar[]`, `circuits[]`, `chart[]`, `results{}`, `driverRounds{}`, `driverStats{}`, `roundCodes[]` — standings and stats **precomputed** via `src/lib/seasonStats.mjs` | 60–120 KB |
| `content.json` | Guide chapters, blog index+bodies (MDX stripped), records leaderboards (from the archive build), trivia facts | ~190 KB |

The apps poll `manifest.json` (cheap, `Cache-Control: max-age=60` via `public/data/app/.htaccess` — that file is static and committed) and re-fetch a season/content file only when its sha256 in the manifest changes. Because the refresh workflow (`refresh-current-season.yml`) rebuilds and deploys the site whenever the current-season bundle changes, **the apps update within minutes of race results landing — no app release needed**.

## Rules — read before changing anything

1. **The `/app/v1/` path IS the schema version.** Within v1, changes must be *additive only* (new fields are fine — the apps ignore unknown keys). Renaming/removing/retyping a field is a breaking change: generate `/app/v2/` alongside v1 and keep v1 alive for a deprecation window.
2. **Season & content files must stay deterministic** — no timestamps or random ordering inside them. Byte-identical output means the FTP deploy skips them and HTTP caching stays effective. Only `manifest.json` carries `generatedAt`.
3. **A feed-generation failure fails the build on purpose.** The FTP deploy deletes remote files missing from `dist/`, so deploying without the feed would delete it from production and strand the apps. Fix the feed rather than bypassing the step. (A single bad season is skipped with a warning; only total failure is fatal.)
4. The reference for the exact season-file shape lives in the app repo: `figures-app/docs/data-contract.md` and `figures-app/design/season-2026.js`. If you change the generator, check the app repo's contract doc and update both sides together.
5. `manifest.json` escape hatches: `notice` (string shown to app users, e.g. planned migration messaging) and `minAppVersion` (per-platform gate). Use sparingly.

## Quick verification

```
npm run build:appfeed     # regenerate just the feed
npm test                  # site tests still green
npm run build             # full site build must pass before pushing to main
```

Sanity-check `public/data/app/v1/seasons/2026.json` after changes: `meta.leader`, a `results` round's `rows[0]`, and `chart[0].pts` length == `meta.completed`.
